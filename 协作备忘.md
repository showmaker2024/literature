# 协作备忘

## 高频事项

- 文献笔记更新：在 `文献摘要.md` 追加对应论文条目，必要时补充配图。
- 提交前检查：先确认本次是否为“完整笔记”还是“部分笔记”，再命名 commit。
- 同步远端：提交后执行 `git push origin main`，并确认本地不再 ahead。

## 提交命名约定

- 文献阅读优先使用：`文献名称 + 笔记`。
- 若内容未完成，使用：`文献名称 + 部分笔记`。
- 禁止使用与实际进度不符的“完整笔记”语义。

## 常见问题与处理

1. 现象：`git push` 报 `HTTP 408` / `unexpected disconnect`。

- 处理：直接重试一次并延长超时。
- 若仍失败：在当前仓库执行 `git config http.postBuffer 524288000` 后再推送。
- 备注：出现 `Everything up-to-date` 但本地仍 ahead 时，以 `git status -sb` 结果为准。

2. 现象：提交/推送命令因提权确认中断。

- 处理：在确认用户意图后，使用同一目标命令重新执行。
- 经验：先对齐提交信息，再发起一次性提权提交，减少等待。

3. 现象：`git status` 中文路径显示转义编码，阅读困难。

- 处理：可在仓库执行 `git config core.quotepath false` 改善可读性。

4. 现象：协作会话中断后，需要“重新连接并更新文档/仓库”。

- 处理：按顺序执行 `git remote -v`、`git pull --ff-only`、`git status -sb`。
- 判定：
  - `pull` 出现 `Fast-forward` 说明已同步到远端最新内容。
  - `status` 显示 `## main...origin/main` 且无变更，说明本地与远端一致。
- 备注：若 `pull` 失败，再按本文件第 1 条的网络重试流程处理。

5. 现象：顶层说明文档出现乱码或仓库用途不清晰。

- 处理：直接将 `README.md` 统一改写为 UTF-8 中文说明，并补充仓库用途、目录结构与维护约定。
- 经验：顶层说明尽量只保留稳定信息，不把一次性的阅读进度写进 `README.md`。

6. 现象：仓库新增了稳定的专题目录或子主题，但说明文档没有同步更新。

- 处理：在 `README.md` 增加“目录导览”一节，优先记录长期保留、且会影响检索路径的目录层级。
- 经验：只写稳定结构，不逐条枚举所有 PDF 文件，避免说明文档频繁失效。

7. 现象：在 PowerShell 中读取 `文献摘要.md` 这类中文 Markdown 时出现乱码。

- 处理：先执行 `$OutputEncoding = [Console]::OutputEncoding = [System.Text.UTF8Encoding]::new()`，再用 `Get-Content -Encoding utf8` 读取。
- 经验：批量整理中文笔记前先确认终端输出编码，否则容易误判文件内容本身有问题。

8. 现象：专题目录改名或新增后，根目录 `README.md` / `文献摘要.md` 仍保留旧名称或失效链接。

- 处理：先用 `Get-ChildItem -Directory` 和 `rg --files -g "*.md"` 核对实际目录与已存在摘要文件，再同步更新两个根文档。
- 经验：根索引只链接真实存在的 `文献摘要.md`；只有 PDF、还没建摘要的目录，单独列到“暂未建摘要”更稳妥。

9. 现象：本地已有未提交改动，但需要先同步 GitHub 最新更新。

- 处理：先执行 `git stash push -u -m "auto-sync-before-pull"`，再执行 `git pull --ff-only origin main`，确认同步后执行 `git stash pop` 恢复本地改动。
- 经验：这种做法适合目录整理类更新，能避免 `pull` 时把未提交的 PDF 增删和笔记改动直接卷进冲突。

10. 现象：文献附带的老 `.mpg` 视频在 Windows 默认播放器中无法播放。

- 处理：先用 `ffprobe` 确认编码；若为旧的 `MPEG-1 video` / `MPEG-PS`，执行 `ffmpeg -i input.mpg -c:v libx264 -pix_fmt yuv420p -movflags +faststart output.mp4` 转为通用 `mp4`。
- 经验：保留原始 `.mpg` 文件，只新增同名或语义化命名的 `.mp4`，更稳妥，也方便后续提交到仓库。

## 复盘习惯

- 每次完成“新类型问题”的定位与修复后，将“现象-原因-处理”补充到本文件。
- 记录以可直接复用的命令为主，避免空泛描述。
